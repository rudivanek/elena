## 1. Product Overview

Elena is a mobile-first web application designed to serve as an AI-powered emotional support companion. The core proposition is a closed loop between three modalities: conversational expression (chat), structured self-reflection (journal), and pattern recognition (weekly insights). The product is positioned as a daily emotional wellness tool, not a clinical intervention.

The target user is someone who wants a private, low-friction space to process emotions — not a user seeking therapy, meditation, or mood gamification. Elena is built for the moment between "I feel something" and "I understand what I feel."

The product runs on a React/TypeScript frontend, Supabase backend (PostgreSQL with Row Level Security), and OpenAI for LLM inference. It is currently deployed on Netlify. All user data is encrypted at rest using a per-user passphrase-derived key. The app is in Spanish, indicating the initial target market is Spanish-speaking.

## 2. Core Features — Mechanics and Purpose

### 2.1 Chat with Elena

The primary surface. Users open a thread and write to Elena, who responds using an OpenAI-powered system prompt. Elena's behavior is shaped by:

- A base persona prompt (supportive, non-clinical, emotionally attuned)
- A dynamic context block injected per request: user memory key-value pairs, recent emotional signal summary, weekly insight prose, conversation mode classification
- A suggestion chip system that presents 2–4 emotionally relevant response prompts after each exchange
- A crisis detection layer (3-stage: regex shortcut → heuristic classifier → LLM fallback) that triggers a crisis resource modal if distress indicators are detected

Threads are persistent. Users can create multiple threads. A `welcome_inserted` flag on the thread table gates a one-time contextual opening message that incorporates the user's most recent weekly insight when the user returns after 48+ hours.

### 2.2 Journal

A private, encrypted journal with separate entries from chat. Features include:

- Free-form entry with optional guided starter prompts generated by an edge function
- Reflection prompts injected into the journal editor via a dedicated AI edge function (`ai-reflection-prompt`)
- Draft state: entries can be saved as drafts before publishing
- A "Convert to Journal" flow that allows a completed chat thread to be saved as a journal entry
- Reverse flow: chat threads can be linked to journal entries via `source_chat_id` and `linked_journal_entry_id` schema fields
- Journal progress tracking: entry count, word count trends, streaks

### 2.3 Weekly Insights

Aggregated emotional intelligence surfaced on a dedicated Insights page. Includes:

- Mood distribution card (emotional tone across the week)
- Streak card (consecutive active days)
- Day-of-week pattern card (which days show higher/lower emotional intensity)
- A `WeeklyInsightPanel` component displaying the auto-generated weekly insight
- A `MiniInsightCard` that surfaces a short AI-generated sentence in the chat flow
- Insights are generated by a `generate-weekly-insights` edge function scheduled via pg_cron

The insight generation pipeline reads from `chat_signal_daily_agg` and `journal_entries`, computes delta scores, and currently outputs template-based prose. An LLM narrative upgrade is in scope but not yet shipped.

### 2.4 Memory System

A user memory store (`user_memory` table) holds key-value pairs that represent persistent facts or themes the user has shared (e.g., "work: high stress lately"). This context is injected into every chat request. A V1 behavioral instruction in the system prompt directs Elena to reference these memories when relevant. Structured keyword matching between incoming messages and stored memory keys is not yet implemented — the LLM makes the connection discretionally.

### 2.5 Emotional Signal Tracking

Every chat interaction is analyzed by a mood detection pipeline that classifies the dominant emotional signal. These signals are aggregated daily into `chat_signal_daily_agg`. The aggregation feeds the insight generation pipeline. Signals track: dominant mood, secondary emotion, topic tags, intensity.

### 2.6 Token Budget System

A daily token budget controls LLM usage per user. Plans are stored in the `profiles` table (`plan_key`). Token consumption is logged per request and rolled up daily. A `get_token_budget` RPC computes remaining budget. When exhausted, an `UpgradeModal` is triggered. No payment processor is currently wired — the upgrade path dead-ends.

## 3. User Experience Flow — First Week

**Day 0 — Signup**
User lands on the marketing page, signs up via email/password. Lands in an empty chat. A first-time detection query checks for zero prior messages and surfaces a minimal welcome block with four soft conversation starters. No onboarding modal, no feature tour, no privacy disclosure visible at this stage.

**Day 0 — First chat**
User types or selects a chip. Elena responds. Mood detection classifies the message silently. Suggestion chips appear after Elena's response. User either continues or closes. No journal or insights are prompted. The user must discover those surfaces independently via bottom navigation.

**Day 1–6 — Return visits**
Each return visit starts a new thread. If the user has not returned in 48+ hours and a weekly insight exists, Elena's opening message references it. Otherwise, a contextual greeting based on time-of-day and prior mood patterns is generated. No external notification exists to pull users back — return is entirely self-motivated.

**Day 7 — First insight**
If the user has had at least a few sessions, the weekly insight generation cron fires. The Insights page populates with mood distribution, streak data, and the auto-generated narrative. The user can navigate there independently; no in-app prompt explicitly announces that insights are available (a subtle banner in chat surfaces this conditionally).

**Ongoing**
The chat-to-journal and journal-to-chat flows are available but not explicitly promoted. The guided reflection prompt in the journal fires when a journal entry is created. Memory accumulates passively. Elena's conversational quality improves as context builds in the system prompt injection.

## 4. Differentiation

### vs. ChatGPT

ChatGPT is stateless across sessions (without paid memory features), generic in persona, and has no emotional signal tracking. Elena maintains a persistent memory context, is persona-locked to emotional support, and accumulates pattern data over time. Elena's chip system also guides the conversation in ways that generic chat cannot.

### vs. Journaling Apps (Day One, Notion)

Traditional journaling apps are passive — they store text. Elena connects the journal to chat expression, injects AI reflection prompts, and surfaces emotional patterns. The cross-linking between chat threads and journal entries is architecturally unique.

### vs. AI Therapy / Mental Health Apps (Woebot, Wysa, Replika)

Woebot and Wysa use CBT scripting and structured conversations. Replika is companionship-focused with no insight layer. Elena sits between: unstructured emotional expression with structured pattern recognition, but without clinical protocols or diagnostic framing. The weekly insight narrative and memory system are designed to create a genuine "this app knows me" sensation that Woebot and Wysa do not produce.

### vs. Mood Trackers (Daylio)

Mood trackers require manual input and produce visualizations. Elena produces insights from natural language without any manual classification from the user.

## 5. Strengths

**Data architecture is solid.** 20 tables, full RLS, encryption at rest, consistent indexing. This is not typical for an early-stage product.

**The loop is architecturally correct.** Express → Reflect → Recognize is a coherent product loop. Chat feeds signals. Signals feed insights. Insights re-enter chat. Few consumer wellness apps have this closed-loop design.

**Crisis detection is real.** A 3-stage pipeline with a crisis resources modal is live. This is a significant liability and trust differentiator relative to competitors who ignore crisis detection entirely.

**Admin tooling is ahead of the curve.** Cost simulator, boundary tests, flight recorder per user, plan limits editor, email lifecycle manager. This allows serious operational oversight and debugging that most early-stage apps do not have.

**Encryption is production-grade.** Per-user passphrase-derived key encryption on all sensitive content. Not typical at this stage.

**Token budget infrastructure is complete.** DB schema, RPC functions, rollup jobs, plan keys, and limits are all in place. Payment wiring is the only missing piece for monetization to go live.

## 6. Weaknesses and Gaps — Honest Assessment

**No monetization path.** The entire revenue model is infrastructure without a transaction. There is no Stripe integration, no upgrade button that works, and no trial-to-paid conversion moment. The token exhaustion screen currently dead-ends.

**Onboarding is absent.** A new user sees an empty chat interface. There is no explanation of what Elena is, what the journal is for, what insights will show, or that Elena is not a licensed therapist. No privacy disclosure is presented at signup. This is both a trust gap and a potential liability.

**Memory is passive.** The memory system stores key-value pairs but does not proactively engage with them in a structured way. Elena can reference memories but relies on the LLM's judgment to decide when. Users who share personal context rarely see it acknowledged back naturally.

**Weekly insight narrative is template-based.** The insight output is generated by delta threshold logic (if mood improved by X, output sentence Y). Users who engage for more than two weeks will begin to notice the mechanical repetition. The narrative layer has not been upgraded to LLM-generated prose.

**No re-engagement outside the app.** Email lifecycle infrastructure is scaffolded and scheduled but delivery is unconfirmed. There are no push notifications. A user who stops opening the app has no external pull back. For a habit-forming wellness product, this is critical.

**Known bugs in production.** The `user-memory` DELETE endpoint silently fails due to an undefined variable. The V1 encryption fallback may produce decryption failures on original content. Week boundary math in `insightWeekly.ts` can be wrong across month boundaries.

**Flight recorder stores full message text.** The QA tooling records complete user message content in `QA_TEMP` payload fields. This is a privacy risk before general availability.

**Overengineered chip system.** Three parallel chip generation systems (`adaptiveChips.ts`, `emotionChips.ts`, `stanceChipPacks.ts`) run with unclear precedence. Adding a new mood state requires updating three files.

## 7. Monetization Model

**Current state:** Partially built, not functional.

The intended model is a freemium daily token budget:
- Free plan: limited daily tokens (exact limit configurable in `plan_limits` table)
- Paid plans: higher daily token budget, potentially additional features

Infrastructure in place:
- `profiles.plan_key` column distinguishes plan tiers
- `token_usage` table tracks per-request consumption
- `token_usage_daily_rollup` aggregates for budget checks
- `get_token_budget` RPC returns remaining tokens
- `admin_plan_limits` RPC allows editing limits per plan

Missing:
- Stripe or any payment processor integration
- A functional upgrade flow (the UpgradeModal exists but has no payment path)
- Email confirmation of subscription state
- Graceful token exhaustion message (currently shows a hard limit error)

The highest-leverage monetization moment — token exhaustion — is currently the worst product moment. Users who have reached maximum engagement hit a wall with no path forward.

## 8. Market Context

The consumer mental wellness market is large and growing but commoditizing at the low end. Generic AI chat (ChatGPT, Claude, Gemini) is free and increasingly capable of emotional support. The differentiation window for a niche emotional AI product requires strong retention mechanics (habit formation, memory, personalization) and a trust positioning that generic chat cannot match.

The Spanish-language market is relatively underpenetrated for premium AI wellness tools. This is a plausible wedge. However, the product's English-language admin interfaces and documentation suggest the language decision may be scope-limited rather than strategic.

The core risk is commoditization before defensibility: if ChatGPT adds persistent memory and persona customization at scale, the differentiation window narrows significantly. The counter-argument is that a purpose-built emotional companion with closed-loop insight generation is a distinct product category — if executed well.

## 10. Key Product Risks

**Revenue gap.** No working payment path. If the product reaches users before monetization is functional, the conversion window closes. Users who experience the product for free develop expectations that are difficult to reset.

**Retention without external re-engagement.** A habit-forming wellness app with no notifications, no emails confirmed-sent, and no streak mechanics has a structural day-7 retention problem. The product depends entirely on intrinsic motivation.

**Trust without clinical clarity.** No visible disclaimer that Elena is not a therapist. If a user in distress relies on Elena as a primary support mechanism and something goes wrong, the product has no documented disclosure. The crisis detection modal partially mitigates this but is not equivalent to a visible "not a therapist" disclosure at first use.

**Privacy before GA.** Flight recorder `QA_TEMP` fields contain full user message text. This is a GDPR and CCPA exposure before public launch.

**Insight quality plateau.** If insight narratives remain template-based, users who engage deeply for 3–4 weeks will disengage when they notice the repetition. The insight system is the long-term retention engine — if it does not improve, long-term retention is structurally limited.

## 11. Future Potential

The architecture supports meaningful upside if execution gaps are closed:

**Narrative insight engine.** Replacing template-based output with LLM-generated personalized weekly narratives would materially change the product experience. The data pipeline already exists. The prompt upgrade is a contained improvement.

**Longitudinal emotional memory.** The current memory system stores facts. A future version could store emotional arcs — not just "high work stress" but "work stress has decreased since March, last major peak was February, correlated with Tuesdays." This would make Elena genuinely relational rather than contextually aware.

**Structured re-engagement loop.** Email + push + in-app nudge based on insight generation events and engagement gaps. The infrastructure exists. The delivery mechanism needs to be confirmed functional and expanded.

**B2B / employer wellness.** The admin infrastructure (cost control, plan limits, user management, token reporting) maps cleanly to an employer wellness or EAP use case. This is not the current positioning but the tooling is ahead of where most consumer products are.

**Cross-session insight feedback.** Allowing users to rate or react to weekly insights would close a feedback loop that currently does not exist. It would also produce training signal for improving insight quality over time.

## 12. Open Questions

1. **Language strategy:** Is Spanish the permanent language of the product, or a first-market decision? The admin and documentation are in English, suggesting the latter — but there is no roadmap confirmation.

2. **Clinical positioning:** The product explicitly avoids clinical framing but has no documented disclaimer or terms visible at signup. What is the intended liability boundary?

3. **Insight quality threshold:** At what point does the insight narrative need to be LLM-generated rather than template-based for the product to deliver its core promise? This is the most important quality question.

4. **Token limit calibration:** The current free tier limit may be too low to allow a complete emotional conversation. If users regularly hit the limit mid-session, the product fails its core use case. What is the intended free-tier experience depth?

5. **Re-engagement confirmation:** Email lifecycle jobs are scheduled but delivery is unconfirmed. Is any email actually reaching users today?

6. **Memory deletion bug:** The `user-memory` DELETE endpoint silently fails. Users who try to delete a memory cannot. Has this been confirmed in production? What is the user-facing experience?

7. **Competitive timing:** ChatGPT's memory feature is expanding. At what point does the differentiation gap between Elena and ChatGPT narrow enough to change the product strategy?

8. **Target user definition:** Is the intended user someone in acute emotional distress, someone practicing ongoing self-reflection, or both? The product currently serves both but does not explicitly design for either.
